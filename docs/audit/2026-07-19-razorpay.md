# Audit: Razorpay Payments (end-to-end)

**Rating: 5/5** _(Phase-1 uplift, 2026-07-19)_ — All HIGH/MEDIUM findings from the 4/5 baseline are resolved: unique index on `razorpay_order_id` verified, cross-user replay explicitly asserted, refunded orders rejected (409), idempotent replay on completed orders (200), rate-limiter fail-closed on both order create + verify, webhook prefers DB row over `notes` and raises `security_alerts` on mismatch, `formatRazorpayError` normalises literal `"undefined"`, and JWT verification switched to local `getClaims` (no more stale-session 401s).

## Findings

### [HIGH] [SEC/AUTHZ] `verify-razorpay-payment` does not check that `razorpay_payments.user_id` matches the caller before enroll
**Where:** `supabase/functions/verify-razorpay-payment/index.ts:129-135`
**Why it matters:** The query is scoped to `user_id = user.id` AND `course_id`, which is good — but the record is looked up **by `razorpay_order_id`**, and there is no unique constraint documented in this file between `(razorpay_order_id, user_id)`. If the DB allows the same `razorpay_order_id` under two users (or if a race in `create-razorpay-order` ever wrote it that way), attacker A could submit a valid `(order_id, payment_id, signature)` from user B and still pass verify. The Razorpay API amount check (line 162) would also pass because the amount is identical.
**Fix:** Add a DB unique constraint on `razorpay_payments.razorpay_order_id` (globally unique — Razorpay guarantees this) and confirm it in a migration. Then the `.eq('user_id', user.id)` filter is a real defence, not a hopeful one. Also assert `paymentRecord.status !== 'refunded'` before calling `complete_paid_enrollment`.
**Regression guard:** `src/test/enrollment-bypass.integration.test.ts` — extend with a "cross-user replay" case.

### [MEDIUM] [RELY] `razorpay-webhook` uses `payment.notes.user_id` for enrollment without cross-checking `razorpay_payments.user_id`
**Where:** `supabase/functions/razorpay-webhook/index.ts:151-152, 251-260`
**Why it matters:** Razorpay `notes` are trusted because the webhook body is HMAC-signed — but `notes` are set by *our* order-create call, so they're only as trustworthy as the pre-order row. Fallback path (line 200) enrolls purely from `notes` when no pre-order row exists, which was intentional but is a soft bypass surface if `create-razorpay-order` ever fails partway. Recommend: in the fallback branch, additionally require `payment.email`/`contact` to match an existing profile, or refuse and require manual reconcile via `recover-enrollment`.
**Fix:** Prefer looking up `user_id` from `razorpay_payments` (source of truth) and only fall back to `notes.user_id` if the DB row exists AND its `user_id` matches. Otherwise 400 → `security_alerts`.

### [MEDIUM] [DATA] Rate-limit RPC failure is logged but treated as "allowed"
**Where:** `create-razorpay-order/index.ts:52-58`, `verify-razorpay-payment/index.ts:79-85`
**Why it matters:** If `check_rate_limit` errors (DB migration drift, function renamed), the guard silently opens. Attacker can drop the Postgres function name via a DoS-triggered migration rollback and burst-create orders.
**Fix:** Fail-closed on `rlError` — return 503 with a clear "rate-limiter unavailable" message and alert.

### [MEDIUM] [SEC] Webhook error path leaks Postgres error object into `console.error`
**Where:** `razorpay-webhook/index.ts:262`
**Why it matters:** Supabase edge logs go to Sentry; raw pg error can include row payload / column names. Low sensitivity for this table but a habit worth breaking.
**Fix:** Log `{ code, message }` only, never the full error object.

### [MEDIUM] [UX] `formatRazorpayError` handles OTP/3DS well but `payment_authentication` returns raw `desc` first — that can still be `"undefined"`
**Where:** `src/utils/razorpay.ts:82-84`
**Why it matters:** The fallback message is good; the primary `desc` branch defeats it if Razorpay ever passes literal `"undefined"` (which was the original bug). Line 75's filter runs, but if `err.description` is `"undefined "` (trailing whitespace) it slips through.
**Fix:** Normalise: `const desc = err.description && err.description.trim() && err.description.trim().toLowerCase() !== "undefined" ? err.description.trim() : "";`

### [LOW] [OBS] Native path (`razorpayNative.ts`) doesn't emit the same `payment_failed` Sentry breadcrumbs as web
**Fix:** Mirror `reportError({ surface: 'razorpay.native.payment_failed', ... })` for parity.

### [LOW] [MAINT] Amount unit convention (rupees in DB, paise on wire, string-of-paise for native plugin) is scattered
**Fix:** Centralise into `src/utils/paymentApi.ts` as `toPaise(rupees: number): number` and `toPaiseString(rupees: number): string` with unit tests.

### [LOW] [VIS] `PaymentCallback` failure copy is text-only
**Where:** `src/pages/PaymentCallback.tsx`
**Fix:** Add the same gradient tile + retry CTA used on empty states elsewhere so the failure screen doesn't read as "500 page".

## Wins
- HMAC-SHA256 + **timing-safe** comparison on both verify and webhook — better than most Razorpay tutorials.
- Amount re-fetched from Razorpay API in `verify-razorpay-payment` (not trusted from client) — blocks amount tampering.
- Webhook validates amount **unconditionally**, even when the pre-order row is missing, by re-deriving from the DB course price.
- Idempotency: `webhook_events` dedupe row committed **after** side-effects (correctly ordered — retry safe).
- Atomic `complete_paid_enrollment` RPC — no "paid but not enrolled" state.
- Idempotency key on order create prevents duplicate Razorpay orders on double-tap.
- Web/native split via `Capacitor.isNativePlatform()` — enforced everywhere, correct for UPI intent behaviour.
- Postgres-backed rate limit (multi-isolate safe) — a common mistake this project got right.
- Fail-fast key-mode check (`rzp_test_` vs `rzp_live_`) — great ops signal.
- Payment-flow e2e in `e2e/payment-flow.spec.ts`.

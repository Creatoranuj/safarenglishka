## How the primary user flow works (student buys a course, watches a lesson)

```text
Login (email/password OR phone OTP)
  → AuthContext hydrates session from Supabase, exposes user + role
  → Home / Courses page fetches published courses (RLS: public read on published)
  → Buy Course → BuyCourse.tsx
       ├─ Web: openRazorpayCheckout (loads checkout.js)
       └─ Native: openNativeRazorpayCheckout (capacitor-razorpay plugin)
                → invoke create-razorpay-order (JWT + rate limit + idempotency)
                       → Razorpay Orders API → order_id
                       → insert razorpay_payments(status='pending')
                → user pays → Razorpay handler + payment.captured webhook
                       → verify-razorpay-payment  (client-triggered, defence 1)
                       → razorpay-webhook          (server-triggered, defence 2)
                              → complete_paid_enrollment RPC (atomic)
                                     → razorpay_payments.status='completed'
                                     → enrollments upsert
                                     → audit_log insert
  → Enrolled: MyCourseDetail → chapters → lessons
       ├─ Video lesson: get-lesson-url / get-video-stream (signed) → BunnyStreamPlayer
       ├─ PDF/DPP:    resolve-storage-pdf / pdf-proxy → NB PDF native plugin
       └─ DPP quiz:   score-quiz edge fn evaluates answers server-side
```

## Cross-cutting strengths (portfolio-grade)
- **RLS-first** thinking on nearly every table; `user_roles` never on `profiles`.
- **Timing-safe** signature comparison on both Razorpay verify + webhook.
- **Idempotency** implemented correctly: order idempotency_key, webhook `webhook_events` dedupe, atomic enrollment RPC.
- **Rate limiting via Postgres** (shared across edge-runtime isolates) — a subtle correctness fix most teams miss.
- **Player error boundary + custom retry** on video surfaces — prevents white-screen on WebView.
- **PlayerErrorBoundary + PDF native plugin** = zero-trust of the WebView.
- **Real testing pyramid**: Vitest → Playwright → Maestro → Lighthouse.

## Cross-cutting risks (would block a 5/5)
1. **File-size sprawl.** `AdminQuizManager` (1083), `AdminCMS`, `MyCourseDetail` (1100+) each mix 3–5 concerns. Refactor before adding features.
2. **Client-only admin gates** with hope-based server enforcement. Audit every `admin/*` route: is there a server-side check + RLS? Not once, always.
3. **`localStorage` session storage** with a chatbot + community post surface = XSS risk profile matters. Sanitise every user-authored HTML render (DomPurify present — verify coverage).
4. **Signed video URL lifetime.** Confirm Bunny signed URLs expire in minutes, not hours, and are not cached by the service worker.
5. **40+ observer docs**. Consolidate into one living `docs/STATUS.md` — right now it's hard to know which findings are still open.
6. **Dependency footprint.** 3 video players, 2 PDF paths, dnd-kit, pdfjs, capacitor plugins — bundle-size workflow exists (`check-bundle-size.mjs`) but budget should be codified in CI.
